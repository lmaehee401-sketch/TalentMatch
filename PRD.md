## 一、产品概述

### 1.1 产品定位

TalentMatch 是一款 **AI 驱动的招聘全链路效率工具**，面向 HR 从业者和招聘团队，通过大语言模型的语义理解能力，将招聘中最耗时、最依赖经验的三个环节——简历匹配、候选人排序、流程诊断——进行智能化改造。

### 1.2 一句话价值主张

**不是用 AI 替代 HR，而是让每一位 HR 都拥有一流的招聘分析能力。**

### 1.3 产品形态

- **类型**: 纯前端单页 Web 应用（SPA）
- **部署**: GitHub Pages（免费静态托管）
- **API**: 调用 DeepSeek Chat API（用户自备 Key）
- **技术栈**: HTML + CSS + JavaScript（零框架依赖）

## 二、问题定义

### 2.1 核心痛点

| 痛点 | 现状 | 影响 |
|------|------|------|
| **语义失配** | 传统筛选依赖关键词匹配（如"3年Java"），无法识别「用分布式锁解决超发问题」=「有分布式系统设计能力」 | 漏掉高潜力候选人，招到"简历漂亮但能力不符"的人 |
| **标准漂移** | HR 连续看 50 份简历后，评判标准不自觉发生变化 | 前 20 份和后 30 份的筛选一致性差 |
| **复盘靠感觉** | 招聘漏斗分析依赖 Excel 手工统计，"技术二面通过率为什么低"只能靠猜 | 问题发现滞后，优化方向模糊 |
| **信息断层** | 简历筛选 → 面试 → 录用决策之间缺乏结构化信息传递 | 面试官拿到的只有一份简历，不知道"为什么推荐这个人" |

### 2.2 目标用户画像

| 用户类型 | 典型场景 | 核心需求 |
|----------|---------|----------|
| **HR 招聘专员** | 日均处理 50-200 份简历，需要快速筛选 | 批量对比，排出优先级 |
| **HRBP / COE** | 负责人才盘点和招聘流程优化 | 漏斗诊断，数据驱动决策 |
| **业务面试官** | 收到 HR 推送的简历，需要高效面试 | 匹配报告 + 追问指南，面出深度 |
| **求职者** | 投简历前自我评估 | 模拟匹配，发现差距 |

## 三、产品架构

### 3.1 三大模块

```
┌─────────────────────────────────────────────────┐
│                  TalentMatch                     │
├───────────────┬───────────────┬─────────────────┤
│   📋 单份匹配   │  📊 批量对比   │   📈 漏斗诊断    │
│  (深度分析)    │  (候选人排名)  │   (流程优化)     │
├───────────────┼───────────────┼─────────────────┤
│ JD + 简历      │ JD + 多份简历  │ 漏斗各环节数据   │
│     ↓         │     ↓         │      ↓          │
│ 四步流水线     │ 逐份匹配+排序  │ AI 瓶颈诊断      │
│     ↓         │     ↓         │      ↓          │
│ 匹配报告       │ 排行榜+详情    │ 诊断报告+建议    │
│ + 面试追问     │ 点击展开       │ + 漏斗可视化     │
└───────────────┴───────────────┴─────────────────┘
```

### 3.2 模块间关系

- **单份匹配** → 深度分析一个候选人与岗位的匹配度
- **批量对比** → 同一岗位多个候选人的横向比较（复用单份匹配流水线）
- **漏斗诊断** → 招聘流程层面的宏观分析（独立模块，互补关系）

## 四、功能详细说明

### 4.1 模块一：单份深度匹配

#### 4.1.1 输入
- **JD 文本**: 支持粘贴、上传 Word/PDF/图片（OCR）
- **简历文本**: 同上

#### 4.1.2 四步分析流水线

| 步骤 | 名称 | 说明 | 关键技术点 |
|------|------|------|-----------|
| Step 1 | JD 结构化解构 | 提取硬性条件、软性能力指标、加分项 | 将模糊 JD 翻译为可验证的行为指标 |
| Step 2 | 简历结构化解构 | 提取基础信息、能力标签、行为证据 | 区分"主导"与"参与"，不编造信息 |
| Step 3 | 语义匹配评分 | 硬性核验(40分) + 软性匹配(40分) + 经验相关度(15分) + 成长性(5分) | 核心差异：语义匹配而非关键词匹配 |
| Step 4 | 面试追问生成 | 基于匹配结果的 STAR 追问 | 风险项→验证排除，亮点→确认深度 |

#### 4.1.3 输出：评估报告
- 综合评分（0-100）+ 环形可视化
- 判定结论：推荐面试 / 可考虑 / 暂不匹配
- 硬性条件逐项核验表格
- 软性能力语义匹配详情（含简历原文证据引用）
- 面试追问指南（口语化追问 + 追问逻辑说明）

### 4.2 模块二：批量对比

#### 4.2.1 输入
- **JD 文本**: 一份（所有候选人共用）
- **简历文本**: 多份，用 `---` 分隔

#### 4.2.2 处理流程
```
JD 解析（1次）
    ↓
逐份简历处理（N次循环）:
  简历解析 → 语义匹配
    ↓
按综合评分降序排列
    ↓
排行榜 + 点击展开详情
```

#### 4.2.3 输出
- 排行榜表格：排名、姓名、分数、判定、学历、年限
- Top 3 特殊视觉标识（金/银/铜色）
- 点击每行展开：硬性条件 + 语义匹配详情（懒加载）
- 限制：单次最多 20 份简历

### 4.3 模块三：漏斗诊断

#### 4.3.1 输入
- **漏斗环节数据**: 每个环节的名称、进入人数、通过人数（转化率自动计算）
- **补充信息**（可选）: 岗位类型、行业、薪资范围等上下文

#### 4.3.2 AI 诊断框架
| 分析维度 | 说明 |
|---------|------|
| 转化率异常检测 | 对比行业基准范围 |
| 瓶颈定位 | 找出转化率最低的 2-3 个环节 |
| 根因推测 | 基于 HR 最佳实践推测可能原因 |
| 改进建议 | 具体可操作的建议 + 预计改善效果 |

#### 4.3.3 输出
- 漏斗可视化柱状图
- 瓶颈诊断卡片（严重/警告/良好三级色标）
- 可能原因 + 改进建议列表
- 总结 + 优先级最高的行动项

## 五、技术架构

### 5.1 整体架构

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   用户浏览器   │────→│  DeepSeek API │────→│  GitHub Pages │
│  (SPA 应用)   │←────│  (AI 推理)    │     │  (静态托管)   │
└──────────────┘     └──────────────┘     └──────────────┘
     │                                             │
     │  localStorage + Cookie                      │
     │  (API Key 存储，不上传)                       │
     └─────────────────────────────────────────────┘
```

### 5.2 技术选型理由

| 选择 | 理由 |
|------|------|
| **纯前端** | 零服务器成本，GitHub Pages 免费托管 |
| **用户自备 API Key** | 安全边界最小化，Key 不经过任何中间服务器 |
| **DeepSeek** | 性价比极高（新用户免费 500 万 token，每次分析约 8000 token） |
| **localStorage + Cookie 双存储** | 一个被清理时另一个自动恢复 |
| **零框架依赖** | 轻量、加载快、面试时展示"懂底层" |

### 5.3 文件结构
```
TalentMatch/
├── index.html          # 单页应用（~1440 行）
├── PRD.md              # 本文件
├── logo-preview.html   # Logo 方案预览
└── .mcp.json           # MCP 服务器配置
```

## 七、成功度量指标

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| **匹配准确率** | 语义匹配与 HR 主观判断一致性 > 80% | 小样本 A/B 测试 |
| **效率提升** | 批量对比 10 份简历 < 3 分钟 | 计时测试 |
| **使用门槛** | 首次用户 30 秒内完成首次分析 | Demo 数据引导 |
| **面试官满意度** | 追问指南"可用不修改"比例 > 60% | 用户访谈 |
| **页面加载** | < 2 秒（GitHub Pages CDN） | Lighthouse |

## 九、竞品分析简述

| 竞品 | 模式 | 差异 |
|------|------|------|
| **传统 ATS**（北森、Moka） | SaaS，关键词筛选 | TalentMatch 做语义匹配，非关键词 |
| **AI 简历工具**（HireVue、Pymetrics） | 算法评分 | TalentMatch 给出证据引用，可解释性强 |
| **通用 AI 对话**（ChatGPT、Claude） | 通用 LLM | TalentMatch 有结构化的招聘专用流水线 |

**核心差异化**：不是黑盒打分，而是 **「评分 + 证据 + 追问」** 三位一体的可解释输出。
